## 需求描述

调整物品详情界面的售卖界面是否自动关闭：
- **部分售卖**：物品还有剩余数量时，不关闭售卖界面与物品详情界面
- **全部售卖**：物品已没有剩余数量时，自动关闭售卖界面与物品详情界面

---

## 原始问题分析

### 原始逻辑
在 `MainGameScreen.kt` 的仓库Tab中，售卖成功后总是关闭两个弹窗：

```kotlin
onConfirm = { quantity ->
    val success = viewModel.sellItem(itemId, itemType, quantity)
    if (success) {
        showSellDialog = false
        showDetailDialog = false
        selectedItemId = null
    }
}
```

### 原始架构问题
- `selectedItem` 是独立的 `mutableStateOf<Any?>`，与 `selectedItemId` 双重维护
- `sellItem()` 内部通过 `scope.launch` 异步更新 StateFlow
- 售卖成功后手动读取 `viewModel.xxx.value` 获取更新后的物品，存在**时序竞态**（StateFlow 可能尚未更新）

---

## 修改方案

### 方案选型

| 方案 | 描述 | 问题 |
|------|------|------|
| A | 手动构造更新后的对象 | 与 StateFlow 更新存在竞态，状态不一致 |
| B | 延迟等待后读取 StateFlow | 引入不可控延迟，体验差 |
| **C** | **`selectedItem` 改为从 `selectedItemId` + StateFlow 派生** | **利用 Compose 重组机制自动同步，无竞态** |

**采用方案 C**。

---

## 代码变更详情

### 1. `selectedItem` 从手动状态改为 StateFlow 派生

**位置**：`MainGameScreen.kt` 的 `WarehouseTab` 函数内

**变更前**：
```kotlin
var selectedItem by remember { mutableStateOf<Any?>(null) }
var showDetailDialog by remember { mutableStateOf(false) }
var selectedItemId by remember { mutableStateOf<String?>(null) }
```

**变更后**：
```kotlin
var showDetailDialog by remember { mutableStateOf(false) }
var selectedItemId by remember { mutableStateOf<String?>(null) }
val selectedItem: Any? = selectedItemId?.let { id ->
    equipment.find { it.id == id }
        ?: manuals.find { it.id == id }
        ?: sortedPills.find { it.id == id }
        ?: sortedMaterials.find { it.id == id }
        ?: sortedHerbs.find { it.id == id }
        ?: sortedSeeds.find { it.id == id }
}
```

**原理**：
- `selectedItem` 不再是一个独立的状态变量
- 每次 Compose 重组时，从最新的 StateFlow 列表中根据 `selectedItemId` 查找当前物品
- StateFlow 异步更新后，下一次重组自动获取最新数据（包括更新后的 `quantity`）
- 物品被完全移除后，`find` 返回 null，`selectedItem` 自动变为 null

### 2. 售卖确认回调逻辑调整

**变更前**：
```kotlin
onConfirm = { quantity ->
    val success = viewModel.sellItem(itemId, itemType, quantity)
    if (success) {
        showSellDialog = false
        showDetailDialog = false
        selectedItemId = null
    }
}
```

**变更后**：
```kotlin
onConfirm = { quantity ->
    val success = viewModel.sellItem(itemId, itemType, quantity)
    if (success && quantity >= itemQuantity) {
        showSellDialog = false
        showDetailDialog = false
        selectedItemId = null
    }
}
```

**逻辑**：
- `quantity >= itemQuantity`（全部售卖）：关闭两个弹窗，清除选中状态
- `quantity < itemQuantity`（部分售卖）：不关闭弹窗，依赖 Compose 重组自动更新显示数据

### 3. `onViewDetail` 回调简化

**变更前**：
```kotlin
onViewDetail = { selectedItem = warehouseItem.item; showDetailDialog = true }
```

**变更后**：
```kotlin
onViewDetail = { selectedItemId = warehouseItem.id; showDetailDialog = true }
```

### 4. 添加 LaunchedEffect 安全网

```kotlin
if (showDetailDialog) {
    LaunchedEffect(showDetailDialog, selectedItem) {
        if (showDetailDialog && selectedItem == null) {
            showDetailDialog = false
            selectedItemId = null
        }
    }
    selectedItem?.let { item ->
        // ... 弹窗内容
    }
}
```

**作用**：当物品因其他操作被移除导致 `selectedItem` 变为 null 时，自动关闭弹窗，防止显示已不存在的物品。

## 关键设计决策

1. **为什么不用 `remember(selectedItemId, ...)` 包裹 `selectedItem`**
   - `selectedItem` 的依赖（`equipment`, `manuals` 等）已经是 `remember` 缓存的列表，每次重组时这些列表会在 StateFlow 更新后自动刷新
   - 额外的 `remember` 不会带来性能收益，反而增加复杂度

2. **为什么 `LaunchedEffect` 同时监听 `showDetailDialog` 和 `selectedItem`**
   - `showDetailDialog` 变化时触发一次检查，防止弹窗打开瞬间物品已不存在
   - `selectedItem` 变化时触发检查，StateFlow 更新导致物品被移除时自动关闭

3. **部分售卖后 `SellConfirmDialog` 的 `maxQuantity` 如何更新**
   - `maxQuantity` 参数来自 `itemQuantity`，而 `itemQuantity = currentItem?.quantity ?: 0`
   - `currentItem` 从 `equipment` / `sortedPills` 等 StateFlow 派生列表中查找
   - StateFlow 更新后，Compose 重组，`itemQuantity` 自动获取最新值，`SellConfirmDialog` 重组后显示正确的最大数量
