三种重构方案

目标：统一间距/尺寸 + 简化布局结构

A

方案A：设计系统 + 组件基础设施

先建立统一的设计 Token（间距、尺寸、字体层级），再提取共享组件（统一对话框、统一插槽、统一筛选器），最后逐界面迁移。改动最大但最彻底。

优点

  • 一套设计 Token 控制所有尺寸和间距，以后调整只需改一处
  • 消除 10+ 处筛选器和境界筛选的重复代码
  • 所有对话框使用统一的包装器，外观一致
  • 未来新增界面也自动符合规范

缺点

  • 涉及 30+ 个文件的大范围修改
  • 需要仔细测试每个对话框不出现布局异常
B

方案B：轻量统一（最小化抽象）

只统一不一致的间距/尺寸值，不新建组件抽象。将到处分散的 8-20dp 间距、48-70dp 插槽统一为固定常量。保持现有架构。

优点

  • 改动量小，风险低
  • 不需要重构组件结构

缺点

  • 重复代码仍然存在
  • 后续新增界面容易再次偏离规范
  • 没有从根本上解决布局结构不一致的问题
C

方案C：分阶段重构

第一阶段：建立设计系统（Token + 字体层级）。第二阶段：提取共享组件。第三阶段：逐界面迁移。每阶段可独立验证。

优点

  • 每阶段有明确的验证标准
  • 出问题容易定位到具体阶段
  • 可以在中途根据效果调整方向

缺点

  • 总周期较长
  • 中间状态可能有些界面已迁移、有些未迁移
