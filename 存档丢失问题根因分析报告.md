# 存档丢失问题根因分析报告

## 版本信息

- **报告日期**: 2026-04-25
- **分析范围**: 最新版本存档系统完整代码路径
- **涉及模块**: StorageEngine、GameDatabase、ProductionSlotRepository、SavePipeline、SaveLoadViewModel

---

## 一、问题现象

部分用户反馈：玩了几分钟后进行手动存档，所有存档消失。部分用户无此问题。经确认均为最新版本。

---

## 二、根因分析（按严重程度排序）

---

### 🔴 根因 1（极高概率）：`fallbackToDestructiveMigration()` 导致数据库整体销毁

**位置**: `GameDatabase.kt:775`

```kotlin
.addMigrations(MIGRATION_4_5, ..., MIGRATION_13_14)
.fallbackToDestructiveMigration()  // ← 任何迁移失败都会销毁整个数据库
```

**机制**:

Room 在打开数据库时验证 schema。如果出现以下任一情况，`fallbackToDestructiveMigration()` 会**销毁整个数据库**（所有表、所有槽位数据全部删除）：

1. **迁移路径缺失**：当前仅定义了 v4→v14 的迁移。如果用户数据库版本为 v3 或更早，不存在迁移路径，Room 直接销毁数据库重建。
2. **迁移执行异常**：任何 Migration 的 `migrate()` 抛出异常（如 MIGRATION_8_9、MIGRATION_9_10 中复杂的 SQL 操作在特定数据下可能失败）。
3. **迁移后 schema 不匹配**：迁移执行成功但产生的 schema 与 Room Entity 定义不一致。特别是 MIGRATION_11_12 和 MIGRATION_12_13 涉及建表、数据迁移、删表，如果生成的表结构与 Entity 定义有任何差异（列类型、约束、索引），Room 会判定 schema 无效并销毁数据库。

**为何只有部分用户受影响**:

- 取决于用户的历史数据库版本（升级路径不同）
- 取决于用户的数据状态（某些数据可能触发迁移中的异常）
- 取决于设备存储状态（磁盘 I/O 错误可能导致迁移失败）

**为何表现为"手动存档后所有存档消失"**:

- 数据库销毁发生在 app 启动时的 schema 验证阶段
- 用户可能未在主菜单检查存档列表，直接进入游戏
- 手动存档后刷新存档列表时，才首次发现所有旧存档已不存在
- 时序上是"先丢失、后发现"，而非"存档操作导致丢失"

### 🔴 根因 2（高概率）：`ProductionSlotRepository.restoreSlots()` 跨槽位删除生产数据

**位置**: `ProductionSlotRepository.kt:376-383`

```kotlin
suspend fun restoreSlots(slots: List<ProductionSlot>) {
    globalMutex.withLock {
        _slots.value = slots
        cache.invalidate()
        dao.deleteAll()      // ← 无 slotId 参数！删除所有槽位的生产数据
        dao.insertAll(slots) // ← 仅插入当前槽位的数据
    }
}
```

**调用链**:

```
SaveLoadViewModel.loadGame() / loadGameFromSlot()
  → GameEngine.loadData(productionSlots)
    → productionCoordinator.repository.restoreSlots(productionSlots)
      → dao.deleteAll()  // DELETE FROM production_slots（无 WHERE 条件）
```

**影响**:

当用户加载槽位 A 的存档时，`restoreSlots()` 会先删除**所有槽位**的生产数据，然后仅插入槽位 A 的数据。这意味着：

- 槽位 B、C、D 等的生产数据被永久删除
- 下次保存槽位 B 时，`writeAllDataToDatabase()` 会用 `deleteBySlot(B)` 删除槽位 B 的生产数据（已为空），然后写入当前内存中的生产数据（可能为空或属于槽位 A）

**同样存在此问题的方法**:

- `initializeAllSlots()` — 第329行 `dao.deleteAll()`
- `clear()` — 第371行 `dao.deleteAll()`

**对比正确实现**:

`StorageEngine.writeAllDataToDatabase()` 中正确使用了 `deleteBySlot(slot)`：

```kotlin
database.productionSlotDao().deleteBySlot(slot)  // ← 正确：仅删除指定槽位
```

`StorageEngine.delete()` 也正确使用了 `deleteBySlot(slot)`。

### 🟡 根因 3（中等概率）：自动存档与手动存档的竞态条件

**自动存档路径**:

```
SaveLoadViewModel.enqueueAutoSave()
  → savePipeline.enqueue(SaveRequest(slot=AUTO_SAVE_SLOT))
    → SavePipeline.consumeLoop() [异步消费者]
      → storageFacade.save(AUTO_SAVE_SLOT, saveData)
        → StorageEngine.save(AUTO_SAVE_SLOT, data)
```

**手动存档路径**:

```
SaveLoadViewModel.saveGame()
  → storageFacade.saveSyncWithResult(slot, saveData)  // runBlocking(Dispatchers.IO)
    → StorageEngine.save(slot, data)
```

**问题**:

1. `saveLock`（AtomicBoolean）仅控制"入队/执行"阶段的互斥，不控制实际数据库写入的互斥
2. 自动存档入队后立即释放 `saveLock`，实际写入在 `SavePipeline` 消费者中异步执行
3. 手动存档获取 `saveLock` 后同步执行写入
4. 两者可能同时写入数据库（不同槽位），虽然 `SlotLockManager` 提供了槽位级写锁，但 `ProductionSlotRepository` 不使用此锁系统

**具体竞态场景**:

```
时间线：
T1: 自动存档开始写入槽位0的生产数据（通过 StorageEngine.writeAllDataToDatabase）
T2: 用户在此期间加载槽位1的存档
T3: GameEngine.loadData() → restoreSlots() → dao.deleteAll()
T4: 槽位0的生产数据被 deleteAll() 删除
T5: 自动存档完成，但生产数据已被清空
```

### 🟡 根因 4（中等概率）：WAL 模式 + synchronous=NORMAL 无存档后强制 checkpoint

**位置**: `GameDatabase.kt:786`

```kotlin
db.openHelper.writableDatabase.execSQL("PRAGMA synchronous = NORMAL")
```

**影响**:

- WAL 模式下 `synchronous=NORMAL` 意味着每次 commit 后不强制 fsync WAL 文件
- 数据写入后仅存在于 WAL 文件中，需等待 checkpoint 才合并到主数据库
- 自动 checkpoint 间隔为 30 秒，WAL 阈值 10MB
- **存档操作完成后不触发强制 checkpoint**
- 如果 app 在存档后、checkpoint 前被系统杀死（OOM killer、低内存等），WAL 中未 checkpoint 的数据可能丢失
- 对于低内存设备，此风险更高

**对比**: 仅在 `shutdown()` 中执行了 `TRUNCATE` checkpoint，正常运行期间不保证数据持久化。

## 三、问题间歇性出现的解释

| 因素 | 影响 |
|------|------|
| 用户历史数据库版本 | v3 及更早版本无迁移路径，直接触发 destructive migration |
| 设备内存压力 | 低内存设备更容易触发 app 被杀、WAL 数据丢失 |
| 存档使用模式 | 多槽位用户更容易发现跨槽位数据丢失 |
| 自动存档时机 | 自动存档与手动存档/读档的时间重叠是随机的 |
| 迁移数据状态 | 特定数据可能触发迁移 SQL 异常 |

## 四、修复建议

### 根因 1 修复

- 移除 `fallbackToDestructiveMigration()`，改为 `fallbackToDestructiveMigrationFrom(4)`（仅对 v4 之前版本允许销毁）
- 或完全移除，对缺失迁移路径抛出异常而非静默销毁
- 补全 v1→v4 的迁移路径
- 对所有迁移添加 try-catch 和日志，确保迁移失败时不会静默销毁数据

### 根因 2 修复

- `restoreSlots()` 改为接收 `slotId` 参数，使用 `dao.deleteBySlot(slotId)` 替代 `dao.deleteAll()`
- `initializeAllSlots()` 和 `clear()` 同理
- 或者让 `ProductionSlotRepository` 的操作也纳入 `SlotLockManager` 的锁体系

### 根因 3 修复

- 在 `SaveLoadViewModel` 中，手动存档前等待 `SavePipeline` 中正在执行的自动存档完成
- 或在 `GameEngine.loadData()` 调用 `restoreSlots()` 时，确保没有正在进行的存档操作

### 根因 4 修复

- 在 `StorageEngine.save()` 成功后，对当前槽位执行 `PRAGMA wal_checkpoint(PASSIVE)`
- 或在 `performFullTransactionSave()` 事务提交后添加 checkpoint
