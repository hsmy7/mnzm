# 宗门战争系统重构报告

## 版本

v2.5.30

## 变更概览

本次对宗门战争系统进行了全面检查、修复和功能补充，主要涵盖以下方向：

- AI攻击流程完整修复（战斗失败销毁队伍、胜利返回出发地、占领驻守）
- 玩家进攻奖励机制（按宗门等级决定品阶和灵石）
- 玩家防守失败惩罚调整（40%物品和灵石）
- 玩家队伍返回移动机制（不再瞬间回城）
- 代码复查发现的严重问题修复

---

## 一、AI攻击流程修复

### 修复前问题

| 场景 | 修复前行为 | 问题 |
|------|-----------|------|
| AI战斗胜利+占领 | 驻守被占领宗门 | 无问题 |
| AI战斗胜利+未占领 | 设为"returning"状态 | `processAIBattleTeamMovement`只处理"moving"状态，返回中的队伍永远无法到达 |
| AI战斗失败 | 设为"returning"状态 | 按需求应销毁队伍，而非返回 |

### 修复后行为

| 场景 | 修复后行为 |
|------|-----------|
| AI战斗胜利+占领 | 驻守被占领宗门（不变） |
| AI战斗胜利+未占领 | 设为"returning"，新增返回移动逻辑，到达出发地后销毁 |
| AI战斗失败 | 直接销毁队伍（status="completed"） |
| AI战平 | 直接销毁队伍 |

### 代码变更

**AISectAttackManager.kt**

- `updateAIBattleTeamMovement` 新增 "returning" 状态处理分支，计算从防守方宗门返回攻击方宗门的移动进度
- 新增 `isTeamReturned()` 方法判断AI队伍是否已返回
- `executeAISectBattle` 修复 `canOccupy` 判定逻辑（见第四节）

**CultivationService.kt**

- `processAIBattleTeamMovement` 新增对 "returning" 状态的处理，返回到达后自动从 `aiBattleTeams` 中移除
- `triggerAISectBattle` 重写战斗结果处理逻辑：
  - 攻击方胜利+可占领 → 驻守（不变）
  - 攻击方胜利+不可占领 → 设为 "returning" 返回出发地
  - 攻击方失败/平局 → 直接销毁队伍

## 二、玩家奖励机制

### 奖励规则

| 场景 | 物品数量 | 灵石计算 |
|------|---------|---------|
| 进攻胜利（未占领） | 随机20-60种 | 物品数 × 宗门等级灵石单价 |
| 占领宗门 | 随机80-130种 | 物品数 × 宗门等级灵石单价 |
| 进攻失败 | 无奖励 | 无 |

### 宗门等级对应品阶和灵石

| 宗门等级 | 品阶范围 | 灵石单价（每种物品） |
|---------|---------|-------------------|
| 小型宗门（level=0） | 凡品-灵品（rarity 1-2） | 2,000 |
| 中型宗门（level=1） | 灵品-玄品（rarity 2-4） | 6,000 |
| 大型宗门（level=2） | 宝品-地品（rarity 3-5） | 30,000 |
| 顶级宗门（level=3） | 玄品-天品（rarity 4-6） | 80,000 |

### 奖励物品类型

每种物品随机从以下6类中均匀选择：

1. 装备（EquipmentStack）- 使用 `EquipmentDatabase.generateRandom(minRarity, maxRarity)`
2. 功法（ManualStack）- 使用 `ManualDatabase.generateRandom(minRarity, maxRarity)`
3. 丹药（Pill）- 使用 `ItemDatabase.generateRandomPill(minRarity, maxRarity)`
4. 材料（Material）- 使用 `ItemDatabase.generateRandomMaterial(minRarity, maxRarity)`
5. 草药（Herb）- 使用 `HerbDatabase.generateRandomHerb(minRarity, maxRarity)`
6. 种子（Seed）- 使用 `HerbDatabase.generateRandomSeed(minRarity, maxRarity)`

物品可重复获得。

### 代码变更

**AISectAttackManager.kt**

- 新增 `SectWarRewardConfig` 数据类，定义品阶范围和灵石单价
- 新增 `getSectWarRewardConfig(sectLevel)` 方法，按宗门等级返回奖励配置
- 新增 `generateWarRewards(sectLevel, itemCount)` 方法，生成随机奖励
- 新增 `WarRewards` 数据类，包含灵石和6类物品列表

**CultivationService.kt**

- `executePlayerBattleTeamBattle` 在进攻胜利和占领时调用 `generateWarRewards` 并通过 `applyWarRewards` 入库
- 新增 `applyWarRewards(rewards)` 方法，将奖励灵石和物品添加到玩家背包

## 四、canOccupy判定逻辑修复

### 修复前问题

`executeAISectBattle` 中 `canOccupy` 的判定逻辑：

```kotlin
val allDefenderDisciples = defenderDisciples.filter { it.isAlive }
val highRealmAllDead = allDefenderDisciples.filter { it.realm <= 5 }.isEmpty()
```

`defenderDisciples` 中的弟子 `isAlive` 状态未因本次战斗更新，导致本次战斗中阵亡的高境界弟子仍被视为存活，`canOccupy` 永远无法为 `true`。

### 修复后

```kotlin
val allDefenderDisciples = defenderDisciples.filter { it.isAlive && it.id !in deadDefenderIds }
val highRealmAllDead = allDefenderDisciples.filter { it.realm <= 5 }.isEmpty()
```

排除本次战斗中阵亡的弟子后再判断化神及以上弟子是否全灭。

## 五、玩家队伍返回移动机制

### 修复前问题

- 进攻胜利后队伍直接设为 "idle"（瞬间回城）
- 进攻失败后队伍也直接设为 "idle"
- 驻守队伍"返回"按钮功能完全失效（检查条件与驻守状态矛盾）

### 修复后行为

| 场景 | 行为 |
|------|------|
| 进攻胜利（未占领） | 设为 "returning"，需移动时间返回 |
| 进攻失败/平局 | 设为 "returning"，需移动时间返回 |
| 占领宗门 | 驻守被占领宗门 |
| 驻守队伍召回 | 设为 "returning"，需移动时间返回 |

### 代码变更

**CultivationService.kt**

- `processPlayerBattleTeamMovement` 新增 "returning" 状态处理分支
- 返回时保留 `targetSectId`（不清空），用于计算起始位置，确保线性插值
- `executePlayerBattleTeamBattle` 中胜利/失败均设为 "returning" 而非 "idle"

**BattleViewModel.kt**

- `returnStationedBattleTeam()` 完全重写：
  - 移除 `isAtSect` 和 `isIdle` 的前置检查
  - 改为检查 `isStationed` 状态
  - 设置 `status = "returning"`、`isReturning = true`、正确的坐标和移动参数
