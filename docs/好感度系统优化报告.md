# 宗门好感度系统优化报告

**版本**: v2.3.26
**日期**: 2026-04-23

---

## 一、优化背景

### 1.1 原系统核心问题

| 问题 | 影响 |
|------|------|
| 好感度增长采用纯百分比公式 (`currentFavor * pct / 100`) | 低好感度时增长极慢，高好感度时反而更快，体验倒挂 |
| 提升好感度途径单一（仅送礼） | 玩家缺乏策略选择，游戏深度不足 |
| 好感度上限100无分层 | 90-100区间拥挤，边际收益急剧下降 |
| 结盟门槛90过高 | 玩家需要大量资源才能达到，挫败感强 |
| 解除结盟扣30好感度 | 惩罚过重，玩家不敢尝试结盟 |
| 好感度衰减3年才扣1点 | 几乎无衰减，刷满后永久不管 |

### 1.2 本次优化范围

本次优化聚焦于**好感度数值体系**本身，不涉及新增互动途径（如联合探索、弟子交流等），这些将在后续版本中迭代。

---

## 二、优化方案

### 2.1 好感度增长公式重构（方案A）

**旧公式**:
```
favorIncrease = currentFavor * percentage / 100
```

**新公式**:
```
favorIncrease = baseFavor + (currentFavor * percentage / 100)
```

#### 基础值配置

**灵石送礼档位基础值**:

| 档位 | 名称 | 灵石消耗 | 基础好感度 |
|------|------|---------|-----------|
| 1 | 薄礼 | 20,000 | +2 |
| 2 | 厚礼 | 200,000 | +5 |
| 3 | 重礼 | 800,000 | +10 |
| 4 | 大礼 | 4,000,000 | +15 |

**物品稀有度基础值**:

| 稀有度 | 名称 | 基础好感度 |
|--------|------|-----------|
| 1 | 凡品 | +1 |
| 2 | 灵品 | +2 |
| 3 | 宝品 | +5 |
| 4 | 玄品 | +8 |
| 5 | 地品 | +12 |
| 6 | 天品 | +15 |

#### 公式效果对比

以好感度10、送厚礼给小型宗门为例：
- 旧：10 × 40% = **4点**
- 新：5 + 10 × 40% = **9点**

以好感度80、送厚礼给小型宗门为例：
- 旧：80 × 40% = **32点**
- 新：5 + 80 × 40% = **37点**

**关键改进**: 低好感度时有了"保底收益"，不再出现送大礼只加1-2点的尴尬情况。

---

### 2.2 好感度衰减机制调整

**旧机制**:
- 3年不送礼扣1点
- 无阈值限制（任何好感度都会衰减）

**新机制**:
- 1年不送礼扣1点
- **仅对好感度 > 80 的关系生效**
- 好感度 ≤ 80 不扣除

**设计意图**:
- 高好感度代表高期待，维护成本高，符合直觉
- 80是结盟门槛，结盟后需要持续投入维护关系
- 低好感度不衰减，避免玩家陷入"越不送礼越冷淡"的死亡螺旋

---

### 2.3 结盟门槛降低

**旧门槛**: 好感度 ≥ 90

**新门槛**: 好感度 ≥ 80

**配套调整**:
- 解除结盟不再扣除好感度（原扣30点）
- 解除结盟仅扣除灵石（结盟费用的10%）

**设计意图**:
- 降低试错成本，鼓励玩家尝试结盟
- 结盟后需要通过持续维护（送礼、共同活动）来保持关系
- 解除结盟的惩罚从"关系破裂"改为"经济损失"，更符合修仙世界观

#### 3.2.1 灵石送礼新公式

```kotlin
// DiplomacyService.kt
giftSpiritStones()
val baseFavor = tierConfig.baseFavor
val favorIncrease = if (percentage != null) {
    val percentageIncrease = currentFavor * percentage / 100
    val adjustedIncrease = ((baseFavor + percentageIncrease) * preferenceMultiplier).toInt()
    if (adjustedIncrease == 0) 1 else adjustedIncrease
} else {
    (baseFavor * preferenceMultiplier).toInt().coerceAtLeast(1)
}
```

#### 3.2.2 物品送礼新公式

```kotlin
    if (currentFavorValue == 0) {
        baseFavor
    } else {
        baseFavor + (currentFavorValue * favorPercentage / 100).coerceAtLeast(0)
    }

    baseFavor

val adjustedFavor = (baseFavorIncrease * preferenceMultiplier).toInt().coerceAtLeast(1)
```

#### 3.2.3 好感度衰减

```kotlin
// CultivationService.kt
private fun processFavorDecay(currentYear: Int) {
    val data = currentGameData
    val playerSect = data.worldMapSects.find { it.isPlayerSect } ?: return

    val updatedRelations = data.sectRelations.map { relation ->
        val involvesPlayer = relation.sectId1 == playerSect.id || relation.sectId2 == playerSect.id
        if (!involvesPlayer) return@map relation

        if (relation.favor <= GameConfig.Diplomacy.FAVOR_DECAY_THRESHOLD) return@map relation

        val yearsSinceGift = currentYear - relation.lastInteractionYear
        if (yearsSinceGift < GameConfig.Diplomacy.FAVOR_DECAY_NO_GIFT_YEARS) return@map relation

        val newFavor = (relation.favor - GameConfig.Diplomacy.FAVOR_DECAY_AMOUNT)
            .coerceAtLeast(GameConfig.Diplomacy.FAVOR_DECAY_THRESHOLD)
        relation.copy(favor = newFavor, noGiftYears = relation.noGiftYears + 1)
    }
    // ...
}
```

## 五、旧存档兼容性

### 5.1 数据模型未变

`SectRelation` 数据结构：
```kotlin
data class SectRelation(
    val sectId1: String,
    val sectId2: String,
    var favor: Int = 50,
    var lastInteractionYear: Int = 0,
    var noGiftYears: Int = 0
)
```

所有字段均未增删，旧存档可直接加载。

### 5.2 行为变化对旧存档的影响

| 场景 | 影响 |
|------|------|
| 已有好感度 > 80 的关系 | 下一年度开始会触发衰减（如果1年未送礼） |
| 已有结盟关系（好感度80-89） | 不会自动解除（结盟检查使用新阈值80） |
| 已有结盟关系（好感度 < 80） | 下一年度 `checkAllianceFavorDrop()` 会自动解除 |

**结论**: 无需数据库迁移，旧存档完全兼容。

## 七、后续优化方向

本次优化仅调整了数值体系，以下方向将在后续版本中考虑：

1. **增加非送礼互动途径**: 联合探索、弟子交流、宗门援助等
2. **好感度等级分层**: 0-20敌对、21-40冷淡、41-60中立、61-80友好、81-95信赖、96-100至交
3. **结盟等级系统**: 结盟后通过共同活动提升结盟等级，解锁更多联动机制
4. **宗门专属事件**: 如"共同抵御兽潮"、"联合论道大会"等
